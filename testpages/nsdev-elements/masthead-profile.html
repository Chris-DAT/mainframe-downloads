Setting: Disabling the default masthead greeting

If you have your own user authentication scheme or sign in/out application, you can disable the default masthead greeting. Let me clarify, you must disable the default masthead greeting so you can "greet the user" yourself (you will show the user as signed in or show them as an anonymous user.)

Use the standard v18 config utility to disable the default greeting service by adding this to your page. This will show the user as the default "anonymous" user state, as well as the default "anonymous" drop down menu for that locale's masthead.

			<script>
				IBMCore.common.util.config.set({
					greeting: {
						enabled: false
					}	
				});
			</script>

API: Setting the user state with greeting

The first main API sets the user state; whether they are anonymous or signed in. This API essentially controls the visual cue that the user sees in the masthead. There are 4 options:

  1. User is anonymous.
  2. User is signed in, showing default v18 "signed in" cue.
  3. User is signed in, showing custom image (ex: their profile image).
  4. User is signed in, showing text (ex: their forum handle/shortname).

To use this API, subscribe to the masthead profile ready event, then call the appropriate API with your custom user/data settings.

NOTE: You will probably want to also add the .runAsap(...) option on your subscription in cases where you are doing asynchronous JS. Adding the .runAsap(...) callback option will run your function in the case that the event you subscribed to already happened.

Show/greet the user as "signed in":

Based on the "type", the JS will use the appropriate data to generate the button. Set your own button content, alt content (aria label), image to use and type, whatever is appropriate for your app/site. It is recommended you set them all if possible. Note the setting on line 7.

		<script>
			// Create our user data object to pass to the API.
			someData = {
				linkContents: "MOM",
				linkAlt: "Welcome back",
				backgroundImage: "https://ibmdev.webmaster.ibm.com/i/random/square/",
				type: "default"
			};
				
			// Subscribe to the profile's ready event then call the API when ready.
			IBMCore.common.module.masthead.subscribe("profileMenuReady", "customjs", function(){
				IBMCore.common.module.masthead.showProfileLinkSignedin(someData);
			}).runAsap(function(){
				IBMCore.common.module.masthead.showProfileLinkSignedin(someData);
			});
		</script>

These are the allowed values for the "type". It's pretty obvious what they will show:

  • default
  • image
  • text

Show/greet the user as "anonymous":

This is the default v18 state. This only changes if the user is signed in or you disabled the default greeting and call the API to show the user as "signed in". If you have a dynamic app and have a sign-out feature that doesn't cause a page reload, you can call this API to change the user state from "signed in" to "anonymous". The steps are the same as the other one... set your data object with whatever text you want, then call the "show as anonymous" API:

 
		<script>
			someData = {
				linkContents: "",
				linkAlt: "Sign in or register, blah blah blah"
			};
			
			IBMCore.common.module.masthead.showProfileLinkAnonymous(someData);
		</script>

API: Customizing the drop-down profile menu

The second main API controls the drop-down menu. Once you have determined that the user is anonymous or "signed in", you can then customize the profile drop-down menu.

To modify the profile drop-down menu, call the API telling it what you want to do and pass an array of link objects (link title and URL). Subscribe to the masthead profile ready event (same as above, if you are not adding this in the <head> of your page, you may want to add the .runAsap(...) option) then call the API with your data settings (unless you previously/already subscribed to the event). Note the setting on line 3:

		<script>
			IBMCore.common.module.masthead.editProfileMenu({
				action: "replace",
				links: [
					{
						title: "This is a link ",
						url: "http://adsfa.fdas.com/fas/f/as"
					}
				]
			});
		</script>

These are the allowed values for the "action". It's pretty obvious what they will do:

  • "append" - Add a link(s) to the bottom of the list.
  • "prepend" - Add a link(s) to the top of the list.
  • "replace" - Replace the entire menu list with your link items.
  • "replaceLinkUrl" * - Replace the URL for a special tagged link. Currently only the "sign in" and "sign out" links.

* The "replaceLinkUrl" is special in that it has a slightly different syntax. You only pass the special menu item type and the URL to replace with.

			<script>
				IBMCore.common.module.masthead.editProfileMenu({
					action: "replaceLinkUrl",
					linktype: "signout",
					linkurl: "//www.ibm.com/blah/blah/blah"
				});
			</script>

API: Showing a new notification count

Part of the masthead profiling container includes an allowance for a basic "new notifications" alert/visual cue. If you want to notify the user they have "new notifications" (whatever that means in context of your site/web app), you can use the showNotificationCount API to show a number in the masthead icon.

For example, to show the signed in user that they have 7 new notifications, simply pass the # to the API:

			<script>
				IBMCore.common.module.masthead.showNotificationCount(7);
			</script>

As part of the notification count API, you can optionally also have that number show up in a specified menu item. If you have a menu item with the data-linktype="notifications" attribute on the <li>, the API will automatically append the notification count # that menu item's text.

Official standards & guidelines

ibm.com/standards/web/design/masthead-profile/


What

The masthead profile container is where default IBM profiling and contextual menu links show. It's the little candlestick man icon next to the burger (menu) icon. By default, normal plain regular IBM web pages do not have to do anything. This only applies to sites and apps that have their own authentication scheme (Partnerworld, Developerworks, Knowledge Center, Marketplace, etc).
If you do not work on a site that has their own custom user database or sign in/out/profiling capabilities, there is nothing here for your. Go ahead and stop reading now.


Why

Sometime in the future (years), we will have one true IBM single sign-in authentication system where no matter which one of the 20 million IBM web pages you view, you can be "signed in" and edit your account, profile, etc. But until then... we have a way for you to use this for your specific application profiling needs.


What's new

v17 had the "greeting" service and the sign in/out option in the masthead, however it was only for Web identity. There are now other systems in place and apps where you can be logged into WI, but not logged into the other apps. And vise versa. As well, there is currently no true "sign out".

Recognizing this, and that this will take 1+ years to accomplish across the enterprise, we have setup APIs to allow sites/apps to hook into that profile container and manage it and customize it with whatever authentication system they are using, as well as changing the contextual dropdown menu.

There are several APIs available that allow you to change the user state, customize the menu (in a variety of ways), show notification counts, and change how the signed-in user is displayed.


Rules & guidelines

The rules are simple... use the APIs and do not hack it with custom JS or CSS. This is very important because this will be changing and evolving in the near future (HTML, class names, styles). If you use the APIs as directed we promise uninterrupted service to this profile container. However if you custom code some stuff based on the current html, classes, elements you see in there now, I guarantee your functionality will break.


How

Several APIs are available allowing you to control and customize the masthead profile container:

  • Show/greet the user as anonymous or signed-in.
  • Customize the drop-down menu based on the user's state.
  • Show the user's "new notifications" count.

Hints and tips

When you subscribe to an element's event you are saying "when this event happens run this function". However, if the even already happened your callback will not fire. The .runAsap(...) option on your subscription allows you to say "when this event happens run this function, and if it already happened run this function. It allows you to run the same function in both cases, or run a different function if the event already happened.
