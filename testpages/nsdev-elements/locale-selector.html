Official standards & guidelines

ibm.com/standards/web/design/locale-selector/


What

The dynamically generated locale selector (in the footer) allows the user to select any IBM corporate portal supported locale and goto that country home page. It also doubles as a translated page selector and allow you to let the user goto a different translated version of your page if you have one.


Why

It's not often, but a user might want to goto a different IBM corporate portal site or view a translated version of the page they're on. The locale selector allows them to do that directly from the page they're on and in a consistent place/fashion across every IBM page.


What's new

  • User's don't need to leave the page to select a completely different country home page.
  • The locale for the page you're on is automagically highlighted.
  • The entire list of IBM supported locales and home pages is automatically generated into a select list navigator widget.
  • Doubles as a "translated versions of this page" list and functionality.
  • Uses Google standard alternate hreflang tagging which aids in your SEO and getting visitors to the right local version from search results.

Rules & guidelines

The locale selector is generated automatically by the corporate list of locales. If you have translated versions of a page you must use standard alternate hreflang tagging in order to automatically enable the "translated versions of this page" feature. More information about alternate hreflang on Google SEO.

  • If an item in the locale list matches the lc-cc locale for your <link>, that locale entry will be moved into the top "Translated versions of this page" section and the URL value will be changed to the one you provided.
  • If there is no match and you did not provide a label attribute, nothing happens for that <link> because we don't know what text to use for the item.
  • If you also provide a data-label attribute for an entry, if an item in the locale list matches the lc-cc locale for your <link>, that locale entry will be moved into the top "Translated versions of this page" section and the URL and display text label will be changed to the ones you provided.
  • If you also provide a data-label attribute for an entry and the codes do not match any in the locale list, your custom item will be added to the top section.
  • Language first: If you only provide the two letter language code in the <link>, every item in the list whose language code matches that will be moved to the top section and the URL you provided for that language will by used on all those items.
  • Excluding locale-selector processing: If you want to include alt lang tags for Google, but don't want the locale selector to process it and move the locale to the "translated versions", you can include a data-exclude="true" attribute on the alt lang tag and the locale selector will simple ignore it and not process that language or locale.

How


Standard implementation

To take advantage of the "translated versions of this page" feature, create a <link> tag in the <head> for each translated locale version of the page including the current page locale. This will move the item into the top section of the select list and replace the URL (value) that the option goes to on click.

<link rel="alternate" href="<Your_page_URL>" hreflang="<language_code>-<country_code>" />

Language only implementation

The same as above, except you only put the language code. This means that for every country that has that language as the option, they will all point to this same URL. This will move all locales with the country code the top section of the select list and replace the URL (value) that the option goes to on click. Note that your target page must still contain both a country and language code, so that the masthead/footer, contact module, and other services know what country they should use for localization.

<link rel="alternate" href="<Your_page_URL>" hreflang="<language_code>" />

Providing a custom label

If you also provide a data-label on the <link> the item's display text will be changed to your label value. Example:

<link rel="alternate" href="<Your_page_URL>" hreflang="<language_code>-<country_code>" data-label="<some_display_text>" />

If your codes do not match a supported IBM locale, your item will be added to the top section.


Providing a custom heading

If you want the heading to be more customized than the default "Localized versions of this page", you can use the standard config utility to set a custom "local versions heading" text. For example: 

		<script>
			IBMCore.common.util.config.set({
				localeselector: {
					localVersionsHeading: "Localized SmarterPlanet pages"
				}
			});
		</script>

Advanced usage

After the locale selector has been injected and setup it will publish a JS "ready" event that can be subscribed to.

If you need to bind a custom onchange event, you can subscribing to the locale selector's "ready" event and then remove the default onchange binding:

jQuery("#ibm-footer-locale-selector select").off("change.defaultAction");

If you want to run the default onchange function after you do your own binding/thing (AKA, you just want to run your own code before the default onchange runs): Disable the default binding like above, bind your own onchange, and then simply call the default change function when you're ready:

IBMCore.common.module.localeselector.defaultOnchangeAction();
