Element classes

  • ibm-row-form
    Formats the form to be rows of fields, using standard column classes. Labels are on top of the fields.
  • ibm-row-form--line
    An add-on used in conjunction with the ibm-row-form class, converts text inputs to be just an underline instead of a box.
  • ibm-column-form
    Formats the form with a 190px fixed-width left column for labels and the right column for the fields. Labels are beside the field.
  • ibm-input-group
    Required on the <span> container when it has radios and checkboxes to properly space the group of inputs.
  • ibm-column-field-label
    Optional class when using a column form but are not using the default basic form > p > label structure. This just properly sets the width of the label to match the fixed column.
  • ibm-styled-input
    Styles a stand-alone text input field. Only needed when using a text field and not using one of the form layout classes.
  • ibm-styled-checkbox
    Styles a checkbox field.
  • ibm-styled-radio
    Styles a radio button field.
  • ibm-field-error
    Adds a red shade to the field to indicate there is an error with the user's value.
  • ibm-field-indent
    Add to a text field or textarea to indent it so it lines up with a checkbox or radio button label. Used when your item is a 'other' and you have a field for user input'd option.
  • ibm-item-note
    Used to style note text under a field for either error or informational text.
  • ibm-fullwidth
    Add to a text field to set it to 100% width to make it automatically expand and contract with on the column it's in.

Form types

There are three different types of forms:

  1. Single column form: The name explains it all: One single column of fields. Labels on the left, fields on the right. End of story.
  2. Row form: A way to make fields in rows. The difference from column forms:
    • Labels are on top of the fields.
    • Use standard columns to lay out your form.
  3. Row form - line form: Version of a row form that uses the label like a placeholder text and fields are underlined only. Textareas should not be used.

Based on what form type you use these in, they will display differently, but the HTML is the same for the fields.


Styling text input & textarea without form layouts

Forms are comprised of 2 main things:

  • Layout
  • Fields

The "form" classes above will set and style the layout of the fields. They will also automatically style text inputs and textareas.

In the cases where you only have a text input or textarea and do not need an entire form layout (for example a single search or filter field input), you do not need to include the forms.css and can simply add the ibm-styled-input class to the input or textarea. For example:

Fields widgets


Select lists

Select lists in <form>s are automatically created as a standard select list widget. All standard attributes still apply and the HTML for them is just standard HTML. There is no special data-attribute nor HTML needed, and they use the select2 widget. They can be binded with standard jQuery events such as .on("change").
The search field will automatically be enabled if there are more than 8 options.

Full information can be found on the select list widget element page.

Selected value:

The search field is enabled by default on all select lists. If you want to disable the search/filter for your list, add data-searchenabled="false" to your <select>

All special APIs and options can be found on the select2 options site.


File input/attachment

File input fields require the new "data-widget" attribute for widgets. The implementation is the same as v17 and the file input widget supports multiple file attachments. NOTE: You can add the data-multiple="true" attribute to allow for multiple file uploads but it requires that your server or script that processes the form supports it. When doing this, it converts the field name to an array, and an array of files is sent to the form processor server.

Full information can be found on the file input widget element page.

Official standards & guidelines

ibm.com/standards/web/design/forms/


What

A form on a web page allows a user to enter data that is sent to a server for processing. Forms can resemble paper or database forms because web users fill out the forms using checkboxes, radio buttons, or text fields.


Why

Forms are used to collect information and sometimes you want to collect information.


What's new

The same basic row and column forms are carried over from v17, but a new type of row form (line form) is new for v18. The form widgets are also new (and improved) from the dojo ones allowing for more standard bindings and more flexible API usage.


Rules & guidelines

  • You must include the auxiliary forms.css and forms.js for forms to look and widgets to function properly.
  • You must use proper HTML and have a <label> for every field.
  • If you have a form with a field and the label is not visible (like a single search text field), you can use aria-label on the <input> instead of making a <label> that's hidden.
  • You can use the new html5 fields like; email, number, search, url, etc, but not all are supported cross browser yet. Know what you are using and know the limitations of support.
  • Do not rely on browser default error checking for fields like "email" nor just JS. Always use server-side validation and error checking.
  • Each type of form has a design limitation. Read and understand the limitations before you decide which style form is right for you.
  • Form field must follow proper web and IBM convention to lay out correctly.
  • If you collect Personal Identifiable information (PII = Name, email, phone #, etc) or greater, you must use the Notice & Choice service.
  • If you collect Personal Identifiable information (PII) or greater, you may be subject to a legal, security, privacy, and access control audit at any time by IBM. It is highly suggested you use IWM or an existing certified PII collection application.
  • You cannot use placeholder text instead of a label or aria-label. It is not a substitute.

How

  • The form element must have the appropriate class on it to style it.
  • Form fields are typically made up of a <p>, <label>, <span> and <input> (field). Without those elements, the fields will not render properly. See below for exact code snippet examples.
  • Since forms are highly customized and each one has their own rules, a generic form validator is available, but ultimately the form validation is up to you and your web site and application. Never rely on JS for form validation as JS can simply be turned off in the browser.
