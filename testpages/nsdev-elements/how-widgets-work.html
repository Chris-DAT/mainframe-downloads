v18 has a notion of two types of JavaScript UI elements: modules and widgets.

A module is defined as a UI element that is part of the standard “page shell” that can only be enabled or disabled, and may have an optional setting. These are automated and more tightly controlled than content space widgets because they provide the consistency in the frame or “chrome” of every IBM page. Modules are fixed, specific UI items that are tied to specific HTML elements generated or coded as part of the HTML page shell. Examples of modules include: masthead, footer, profile menu, burger menu, back-to-top link, feedback link, locale selector, etc... View the full list of modules.

A widget is defined as a UI element used in the page's main content space. You choose what widgets to use, what content elements to use them on, and the settings for each instance. Widgets can have many settings and are easily configurable with HTML data-attributes or a JS data object. Examples of widgets include: overlay, tooltip, video player, show/hide, data tables, carousel, etc... View the full list of widgets.

Widgets currently have three main features:

  1. Init event.
  2. Processed class.
  3. Destroy method.

Widgets work by calling their main function which converts the content space element into the widget. As a convenience, v18 automatically runs the widget's function on any element tagged with the data-widget attribute that exists when the page loads (DOM content ready/load event). To use a widget on HTML elements that are dynamically injected after DOM ready, simply run the respective widget's function on the element after you inject it. For example, to dynamically inject HTML and create an overlay widget with it: 

				<script>
					var myHtml = '
This is some text that goes inside an overlay.
';
					jQuery(myHtml).prependTo('#somecontainerid').overlay();
				</script>

When widgets initialize and get created, each widget instance adds the ibm-widget-processed class and fires an ibm-init init event on the DOM element the widget was initialized on. The existence of the class denotes that the widget has already been initialized and any APIs or functions it provides are ready to be used.

If the widget container does not have the ibm-widget-processed class yet, you can subscribe to the widget's ibm-init event it fires on your element so you know when the widget is ready and its APIs are available for use. The custom event is a standard subscribable DOM event that can be binded just like any DOM event:

Note: There are three widgets that fire specially named events instead of ibm-init because they can co-exist with other widgets on the same DOM element:

  • Form validator: fires ibm-formvalidator-init on the form.
  • Table row selector: fires ibm-tablesrowselector-init on the table.
  • Grid size change: fires ibm-gridchange on HTML root element.

				<script>
					// Using native JS.
					document.getElementById("myWidgetContainerID").addEventListener("ibm-init", function (evt) {
						console.log("Widget exists now: ", evt);
					});

					// Using jQuery.
					jQuery("#myWidgetContainerID").on("ibm-init", function (evt) {
						console.log("Widget exists now: ", evt);
					});					
				</script>

For example, if you need to know when a carousel widget has been auto-initialized and is ready so you can use the carousel APIs, you would do something like this:

				<script>
					////////////
					// Using native JS:
					////////////
					
					// This is our carousel container where we used data-widget.
					var myCarouselEl = document.getElementById("myCarouselWidgetElementID");
					
					// If the container has the processed class, the widget already exists and is ready,
					// Else, subscribe to the widget's init event so we get notified when it's ready to be used.
					if (myCarouselEl.className.indexOf("ibm-widget-processed") > -1) {
						// Carousel widget exists, APIs can be used now.	
					}
					else {
						myCarouselEl.addEventListener("ibm-init", function (evt) {
							// Carousel widget exists, APIs can be used now.	
						});
					}
					
					////////////
					// or the jQuery way:
					////////////
					
					var $myCarousel = jQuery("#myCarouselWidgetElementID");
					
					if ($myCarousel.hasClass("ibm-widget-processed")) {
						// Carousel widget exists, APIs can be used now.	
					}
					else {
						$myCarousel.on("ibm-init", function (evt) {
							// Carousel widget exists, APIs can be used now.	
						});
					}
				</script>

To destroy any widget and return the element back to the original HTML that existed when the widget was initialized on it, run the common destroy method found on every element's widget instance: .data("widget").destroy() . Using the example above, to destroy the carousel you would do like this:

				<script>
					jQuery("#myCarouselWidgetElementID").data("widget").destroy();
				</script>
