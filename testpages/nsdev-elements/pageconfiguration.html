All modules IBMCore.common.module.xxxxx have a common convention of dis/enablement. All standard shell page modules can be enabled and disabled the same way. Exactly what options can be changed for modules and what can be enabled and disabled vary per module and are decided upon by the business, executives, design, and governance teams.

How it works is that a default configuration object is set in v18 which contains all of the defaults setting, most of which are configurable by the page JS. There are two ways to set your page's v18 configuration: The standard config utility (same as v17) and through the new digitalData object (DDO).


Configuration utility

A common utility exists that allows you to change the default configurations of a v18 page. When this config utility is called, it simply merges in the object that you pass into the existing default configuration object.

If you have multiple items that you are changing the configuration for, you can call the config utility multiple times or you can bundle them all into one object and call it once. v18 waits until the <head> is loaded and then uses whatever config settings are set to determine how to build the page and what dynamic modules to show/enable. This means your use of the config utility must happen in the <head> otherwise it is too late and v18 has already used the settings that existed.


As a single call

This is typically how you would set all your page configurations; with a single use of config.set

			<script>
				IBMCore.common.util.config.set({
					merchandising: {
						enabled: false
					},
					sharethispage: {
						enabled: true
					},
					masthead: {
						type: "__REPLACE_ME__"
					}
				});
			</script>

As individual calls

You normally wouldn't do this unless you had different scripts or different levels of scripts you controlled on your page and you could not do all your settings in one single call.

			<script>
				IBMCore.common.util.config.set({
					merchandising: {
						enabled: false
					}
				);
				
				IBMCore.common.util.config.set({
					sharethispage: {
						enabled: true
					}
				};
				
				IBMCore.common.util.config.set({
					masthead: {
						type: "__REPLACE_ME__"
					}
				});
			</script>

Setting via the DDO (digitalData object)

Normally you put the v18 JS in the <head> with standard <script> tag. However, v18 is flexible and you can load the v18 JS using @defer, @async or put it at the bottom of the <body> or just async inject it when you want. If the v18 JS is loaded any other method than standard <script> tag in the head, you have to set your page config using the DDO and put your settings in there.

The DDO contains the page's meta data and you can define the config settings in the DDO so that whenever you load in the v18 JS, the settings are there and ready to be used by the JS.

Note that the syntax is the same, and the object that you create is the exact same. It's just that instead of passing it to a function, you are putting it in the DDO in the ibm property/namespace section:

			<script>
				digitalData = {
			        page: {
			            category: {
			                primaryCategory: "____"
			            },
			            pageInfo: {
			                effectiveDate: "____",
			                expiryDate: "____",
			                language: "____",
			                publishDate: "____",
			                publisher: "IBM Corporation",
			                version: "v18",
			                ibm: {
			                    contentDelivery: "____",
			                    contentProducer: "____",
			                    country: "US",
			                    .......
			                    config: {
									merchandising: {
										enabled: false
									},
									sharethispage: {
										enabled: true
									},
									masthead: {
										type: "__REPLACE_ME__"
									}
								}
			                }
			            }
			        }
			    };
			</script>

Official standards & guidelines

ibm.com/standards/web/design/page-configuration/


What

Allows a page to set configuration options on what modules or settings to enable or disable.


Rules & guidelines

Use the config utility. If there's not an option to set or change the config, it's not meant to be changed.
