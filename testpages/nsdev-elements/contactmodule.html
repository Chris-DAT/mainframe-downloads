Official standards & guidelines

ibm.com/standards/web/design/contact-module/


What

A contact module provides an easy and consistent way for users to contact IBM no matter what page they are on.


Why

Users shouldn't have to hunt down where on the page the "contact" information is. It should be persistent across the entire (IBM) site so they always know where to find it without learning a new pattern from page to page or site to site. The contact module and it's standard set of options provides that.


What's new

The design, positioning, and functionality of the contact module is set based upon industry best practices and our own internal testing. The HTML is the same as v17 but with very minor changes. Do not put a Liveperson placeholder in your contact module (it won't work properly), and to make an icon item that is not a link has a new standard class convention.


Rules & guidelines

  • The contact module must be wrapped in <div id="ibm-contact-module"><div>
  • The contact module must not have the Liveperson placeholder container in it like v17 did.
  • The contact module widget injects the button into the #ibm-content-body element. If you do not have this standard required element on your page you will not get the button/trigger link.
  • If you enable Liveperson on your page, it will automatically inject the proper HTML needed above your standard contact module list.
  • You contact module can be coded anywhere on your page inside of the standard grid columns just like normal content. v18 will transform it into the special widget.
  • If you're not sure where to code your contact module, code it somewhere on the page where you would want it to show if the user has JS off, or if you were to have a JS error that prevents the JS from executing.

Case: Full contact module coded in the HTML

If you page is normal and has a contact module coded in the HTML file, there's nothing special you need to do.

Case: #ibm-contact-module doesn't exist, dynamically injected

If you do not have <div id="ibm-contact-module"> coded on your page at all (element doesn't exist) and are dynamically injecting the entire contact module <div id="ibm-contact-module">, you can init the widget after the contact module has been injected by running it's init function:

IBMCore.common.module.contactmodule.init();

Case: #ibm-contact-module exist empty, dynamically injected content

If you have <div id="ibm-contact-module"> coded on your page but it's an empty placeholder and the contents are dynamically injected into it, you will do the same thing as the previous case - You can init the widget after the contact module contents have been injected by running it's init function:

IBMCore.common.module.contactmodule.init();

Case: I don't care about IBM branding or consistency and I don't want the standard contact module button and widget

If you are using a non-standard contact module or design and you do not want the standard contact module widget (and button) at all, you can disable the contact module widget by adding this JS config to your page:

		IBMCore.common.util.config.set({
			contactModuleWidget: {
				enabled: false
			}
		});

What about Liveperson?

If you have a contact module on your page and you want to add the dynamic Liveperson chat service, you can enable Liveperson on your page by adding a couple of settings in the digitalData object and it will automatically show up in your contact module based on the routing rules. View the Liveperson page for more details and full implementation instructions.
